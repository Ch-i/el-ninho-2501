Cap: sniffing a careless dashboard

An IDOR hands you someone else's packet capture, the capture hands you a cleartext FTP login, and a stray Linux capability on the Python binary hands you root. Three small mistakes, one clean chain.

The shape of it

Cap is a study in trusting the client. A Flask "Security Dashboard" lets a logged-in user run a five-second packet capture and download the resulting PCAP. Nothing here is exotic — the whole box turns on three ordinary oversights lined up in a row:

nmap 3 ports → 21/22/80 └─ :80 Flask "Security Dashboard" (gunicorn) └─ IDOR on /data/0 → download someone else's PCAP └─ PCAP holds a cleartext FTP login └─ nathan : Buck3tH4TF0RM3! (reused on SSH) → user.txt └─ getcap: python3.8 = cap_setuid └─ os.setuid(0) → root → root.txt
01

Recon — three doors

I started with RustScan to sweep all 65,535 TCP ports fast, then let it hand the open ones to nmap for service detection. Three ports answer: FTP, SSH, and an HTTP app served by gunicorn — a dead giveaway for a Python web app behind it.

The dashboard greets you as Nathan and exposes a handful of routes: /ip (server-side ifconfig), /netstat, and the interesting one — /capture, which runs tcpdump for ~5 seconds and redirects you to /data/<id>.

02

Foothold — the ID you weren't given

Run a "Security Snapshot" and the browser lands on /data/1, then /data/2, and so on — a plain, sequential, unauthenticated integer. That's the tell for an IDOR (Insecure Direct Object Reference). The captures I triggered were empty (a quiet five seconds of traffic), but nothing stops me from walking the counter backwards to id 0 — a capture someone else made earlier, sitting there with 72 real packets in it.

Takeaway

Any download or report endpoint keyed by a sequential ID is an IDOR until proven otherwise — and the interesting record is rarely the one the app just handed you. Always walk the counter down to 0.

03

Root — a capability that shouldn't be there

First thing on a Linux box after a shell: check file capabilities. getcap -r / turns up something a general-purpose interpreter has no business holding — cap_setuid on the Python binary.

The exploit is a single line — no compiler, no SUID binary, no password:

nathan@cap:~$ /usr/bin/python3.8 -c 'import os; os.setuid(0); os.system("bash")'
# id
uid=0(root) gid=1001(nathan) groups=1001(nathan)
Why it works

cap_setuid grants the ability to call setuid(0) without being root already. Python is an interpreter — it will happily run that call and then execute anything, so the capability is effectively a root shell. (Only cap_setuid is granted, not cap_setgid, so setgid(0) raises PermissionError — drop it and use setuid alone.)

Fixing Cap

  • Authorize the object, not just the route. /data/<id> and /download/<id> must check that the capture belongs to the requesting user; unguessable IDs (UUIDs) are defense in depth, not the fix.
  • Don't ship cleartext FTP. Use FTPS/SFTP, and never seed demo captures with real credentials.
  • Strip stray capabilities. setcap -r /usr/bin/python3.8. An interpreter should never hold a privilege-changing capability.
  • Break password reuse between FTP and SSH.
