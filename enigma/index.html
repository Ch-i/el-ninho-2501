Enigma

A read-only NFS share hands out an onboarding PDF with a webmail password. That mailbox reuses into the next, and the next holds the admin login for an invoicing app that’s one CVE from a shell. From www-data the whole box reuses passwords — a bcrypt falls to bestfriends — and a root-owned OliveTin button runs your command as root.

The shape of it

Seven links, each a small misplaced trust — in an open share, in a reused password repeated four times over, in a filename, and finally in a “safe” button that runs as root:

NFS /srv/nfs/onboarding (world-readable) → New_Employee_Access.pdf └─ webmail kevin:Enigma2024! → Roundcube mailbox └─ password reuse → sarah mailbox → OpenSTAManager admin:Ne3s4rtars78s └─ OpenSTAManager 2.9.8 → CVE-2025-69212 (.p7m cmd injection) → www-data └─ config.inc.php → DB → zz_users haris bcrypt → hashcat → bestfriends └─ SSH is key-only → su haris → user.txt └─ root OliveTin :1337 backup_database → password-arg injection → root
01 · recon
01

An open share and a stack of names

Port 80 is virtual-hosted into mail001 (Roundcube) and support_001 (OpenSTAManager 2.9.8). Alongside the web stack sit Dovecot (IMAP/POP3) and, tellingly, NFS on 2049. showmount reveals a world-readable export — /srv/nfs/onboarding — holding a single onboarding PDF that leaks a new hire’s webmail password.

Takeaway

Mount the NFS export read-only and read it: mount -t nfs -o ro 10.129.239.191:/srv/nfs/onboarding /mnt. Onboarding material is a classic credential seed.

02 · mail reuse
02

One password, four mailboxes

The PDF gives kevin:Enigma2024! for Roundcube. Kevin’s inbox is quiet, but the same password logs in as sarah — reuse. Sarah’s inbox holds an IT provisioning mail for OpenSTAManager at http://support_001.enigma.htb with admin:Ne3s4rtars78s. That’s the intended way onto the invoicing app.

Note

Roundcube stores each session’s IMAP password encrypted with des_key in its session table — recoverable once you have the DB. Here it only ever held kevin and sarah, so the mail path stops at the OpenSTAManager creds.

03 · web RCE
03

A .p7m filename that runs

OpenSTAManager 2.9.8 is vulnerable to CVE-2025-69212. Its electronic-invoice importer passes an uploaded .p7m file to openssl through exec() with the filename inside double quotes — and $( ) is still evaluated there. Because filenames can’t contain /, the command is base32-encoded and decoded at runtime. One authenticated request → code execution as www-data.

# sink — src/Util/XML.php::decodeP7M()
exec('openssl smime -verify -noverify -in "'.$file.'" -inform DER ...');
# filename: z$(echo${IFS}<B32>|base32${IFS}-d|bash).p7m
04 · loot
04

The database gives up a hash

As www-data, config.inc.php yields the MySQL login brollin:Fri3nds@9099, and Roundcube’s config leaks its des_key. More useful: OpenSTAManager’s own zz_users table holds a bcrypt hash for haris — the only real shell user (uid 1000) on the box.

05 · user
05

bestfriends, then su

Known passwords don’t match the hash, so hashcat runs it against rockyou (mode 3200). It falls quickly to bestfriends. SSH refuses passwords for haris (publickey only), but that doesn’t matter — we already have a shell as www-data, so su haris works locally. Drop an SSH key into ~haris/.ssh/authorized_keys for a stable session.

06 · root
06

A button that runs as root

A root process listens on 127.0.0.1:1337 — OliveTin, a web UI that turns configured shell commands into buttons. Its config sets authRequireGuestsToLogin: false with exec: true, so an unauthenticated guest can trigger any action. The Backup Database action builds a shell command from a password-type argument that OliveTin does not sanitize:

shell: "mysqldump -u {{ db_user }} -p'{{ db_pass }}' {{ db_name }} > /opt/backups/backup.sql"
arguments:
  - name: db_pass
    type: password   # free-form → breaks out of the single quotes

Setting db_pass to x'; cp /bin/bash /tmp/rootbash; chmod 4755 /tmp/rootbash; echo ' makes OliveTin (root) copy bash and set it SUID. Fire it via the REST API — no login — then run the SUID shell.

Why it works

Interpolating any user argument into a shell string is injection, full stop — a password type is not a validation. OliveTin should pass arguments as an exec argv array (no shell), enforce argument regexes, and never run guest actions as root.

Closing Enigma

  • Lock down NFS — don’t export onboarding material world-readable; and never ship a live password in a “change it later” PDF.
  • Kill password reuse — one credential opened two mailboxes, an app admin panel, and (via a weak hash) a shell account.
  • Patch OpenSTAManager past 2.9.8; never build a shell string from a filename or any external input.
  • Argv, not shell — OliveTin (and every wrapper like it) must execute actions as an argument vector, validate inputs, require auth, and drop root.
